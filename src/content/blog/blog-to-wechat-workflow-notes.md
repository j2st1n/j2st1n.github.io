这篇算是一次折腾记录。

目标其实很简单：以后写完一篇博客，如果觉得还不错，我希望只说一句话，比如：

> 发布并同步公众号

或者博客已经发了，再补一句：

> 同步到公众号

它就能把最近那篇博客转成一篇微信公众号草稿，封面、正文图片、阅读原文这些都顺手处理好。

听起来不复杂，但真动手的时候，坑还是一层套一层。

## 一开始以为只是“发一下接口”

最开始的直觉是：公众号不是有接口吗？拿 `AppID`、`AppSecret`，调一下草稿接口，事情就完了。

结果很快发现，真正难的从来不是“有没有接口”，而是中间那一大段：

- 博客文章到底从哪里取
- 封面图怎么处理
- 正文图片能不能直接用图床外链
- 博客排版和公众号排版差多少
- “阅读原文”要怎么接
- 用户想要的是“自动发布”，还是“先建草稿”

如果这些不先想清楚，最后做出来的东西就会很别扭：技术上能跑，内容上难看，流程上也不顺手。

## 第一件事：别重造博客发布

博客发布这件事，其实已经有现成的 `bins-blog` workflow 了。

所以这次没有去重写“发布博客”，而是把问题拆成两半：

1. 博客继续按现有 skill 发布
2. 我们只补一个“同步到公众号”的后置动作

这个决策很重要。因为一旦把“博客发布”和“公众号同步”揉成一坨，后面就很难维护。

拆开之后，工作流就清楚了：

- `发布博客` → 还是原来的 blog skill
- `同步到公众号` → 新增的 wechat-sync
- `发布并同步公众号` → 前两步串起来

## 第二件事：得先知道“最近发的是哪篇”

这个问题比想象中关键。

如果用户只说一句“同步到公众号”，系统总得知道要同步哪篇。最开始虽然也可以靠猜：看最近改了哪个 Markdown，或者从 git diff 里抠，但这种办法一看就不稳。

所以后来在 `bins-blog` 里补了一层很轻的状态记录：

- 成功发布后写一个 `last-published.json`
- 里面放这篇文章的：
  - `title`
  - `slug`
  - `url`
  - `description`
  - `cover`
  - `sourceFile`

这样一来，公众号同步就不用猜了。

用户说一句“同步到公众号”，系统默认读取最近一次发布记录，直接知道该拿哪篇文章。

这一步看起来不起眼，但其实是整条 workflow 能不能顺起来的关键。

## 第三件事：阅读原文不该硬塞在正文里

这个也属于一开始想当然的地方。

最早的草稿转换里，我曾经把博客链接直接放进正文，想着至少用户能点过去看原文。

但很快就发现，这很丑。

公众号文章正文里硬塞一段裸链接，看上去就像系统拼出来的，不像一篇正常文章。

后来改成走公众号本身的 `content_source_url` 字段，也就是让博客链接出现在：

> 阅读原文

这样一下就顺了。

正文保持干净，博客链接也还在，而且符合大家对公众号文章的使用习惯。

## 第四件事：封面不是最难的，正文图片才是

这次一开始打通的是封面。

博客文章的 `ogImage` 本来就放在 Cloudflare R2 上，通过 `img.bins.blog` 提供公网访问。最开始的处理方式是：

1. 读取 `ogImage`
2. 从公网 URL 下载到临时文件
3. 再上传到公众号素材库
4. 用它作为草稿封面

这一步其实很快就跑通了。

真正拖后腿的是正文图片。

一开始正文里的图片还是直接沿用博客外链图。技术上不一定立刻坏，但体验上很别扭，而且也不够“公众号原生”。

后来又补了一层：

- 识别正文里的图片
- 逐张下载
- 调微信公众号正文图片上传接口
- 把文章 HTML 里的图片 URL 替换成公众号自己的图

到这一步，草稿终于开始像一篇正经的公众号文章了，而不是“博客 HTML 搬运版”。

## 第五件事：最难受的是排版，不是接口

如果只看“创建草稿成功”这个指标，其实很早就成功了。

但第一次打开草稿时，问题非常明显：

- 能看，但不好读
- 像一坨裸 HTML
- 标题、段落、引用、图片都只是“有”，没有阅读感

于是后面花最多时间的，反而不是 API，而是正文转换器。

大致经历了几轮：

### 第一轮：先能看

先把 Markdown 变成基础 HTML，保证：

- 标题能出来
- 段落不乱
- 图片能显示
- 引用和代码块不至于塌掉

这一版属于“技术上通过”。

### 第二轮：像一篇正常公众号文章

开始补这些：

- 段落行高
- 段间距
- 标题层级
- 图片居中和留白
- 图注样式
- 导语块

这一版的目标不是“惊艳”，而是先摆脱那种“程序自动拼接感”。

### 第三轮：往公众号原生阅读感靠

这一步才开始动“节奏”。

比如：

- 开头的摘要做成导语卡片
- 转折处加一点轻分隔
- `PS` 保留成小标题
- 少量短句做强调，但不过度上价值

这里最容易翻车的地方是：一旦做得太用力，就会有很重的“运营文模板感”。

后面又专门收了一轮，把太硬的切分拿掉，保留结构，但不过分用力。

## 第六件事：默认只建草稿，不直接发

这个几乎没什么好犹豫的。

博客发错了，改一下问题还不算大；公众号如果直接群发翻车，那体验就很差。

所以这条工作流最后定下来的默认行为很清楚：

- 博客可以发布
- 公众号默认只建草稿
- 不自动群发

这其实也是最符合真实使用习惯的。

因为不是每篇博客都值得同步到公众号，也不是每篇都适合原样发过去。

## 现在这条链路已经是什么状态

到目前为止，这套 workflow 已经能做到：

1. 博客发布成功后记录最近文章元数据
2. 读取最近发布文章
3. 转成公众号可接受的 HTML
4. 把博客链接放进“阅读原文”
5. 上传封面到公众号素材库
6. 上传正文图片到公众号正文图片接口
7. 创建微信公众号草稿

而且这条链路已经不是纸面设计，而是实际跑通过多次的。

## 最后怎么收口

最后做的不是继续加功能，而是收口。

因为这种东西最怕“越做越花”。

所以当前 baseline 反而是比较克制的：

- 默认取最近发布文章
- 默认只建草稿
- 默认把博客 URL 放进阅读原文
- 正文图片上传到公众号
- 样式保持有结构、可读，但不过度模板化

对我来说，这样才算真的能用。

不是那种“演示一次很惊艳，但第二篇文章就开始歪”的 workflow。

## 之后最理想的使用方式

如果后面再往前走一步，我希望它最终变成这样：

- 说一句：`发布博客`
- 它正常发博客

然后如果想同步公众号，再说一句：

- `同步到公众号`

它就自动把最近那篇发成公众号草稿。

如果再偷懒一点，也可以直接说：

- `发布并同步公众号`

内部就是先走 blog，再走 wechat-sync。

体验应该是自然的，而不是让我记一堆脚本命令。

## 这次折腾的一个小感受

做这种内容 workflow，真正费劲的从来不是“通不通”，而是“顺不顺”。

接口能调通，只是开始。

真正决定它以后会不会被持续使用的，是这些小地方：

- 文章识别是不是稳
- 阅读原文放得对不对
- 图片是不是像原生内容
- 排版是不是让人愿意看
- 默认行为是不是足够克制

这些细节如果没磨好，这条链路就算 technically works，也很难真的进入日常使用。

而这次总算是把它磨到了“可以拿来真用”的状态。
