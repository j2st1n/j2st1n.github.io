## 我用龙虾搭了个静态博客

最近我做了一件放在以前，可能要折腾很久的事：搭了个静态博客。

如果按过去的办法来，这事通常不会太轻松。要选框架，要挑主题，要接部署，要处理图片、域名、分享卡片和样式。单看每一步都不算特别复杂，但真正麻烦的，是你得先知道整条路怎么走，再自己一点点把它接起来。

这次不太一样。

我用 OpenClaw，一点点把这个博客搭起来了。更准确一点说，是把原来要折腾很久的一整套搭建过程，慢慢压缩成了越来越接近一句话的东西。你告诉它你想要一个什么样的博客，想要什么风格，想怎么部署，想怎么发图，它就开始帮你把后面那些零碎步骤接起来。

博客最后跑起来的时候，我一个很直观的感受是：原来很多事情，不是做不到，而是以前从“想做”走到“做成”，中间那段技术摩擦太长了。现在这段路还在，但已经没有以前那么难迈过去了。

## 我做了个博客发布 skill

博客搭起来以后，我很快发现，真正会反复出现的，不是“搭建”这件事，而是“发布”。

因为静态博客最麻烦的地方，不是第一次建站，而是后面每次发文。写文章、配图、处理文章开头那段元信息、构建、提交、推送、检查部署结果……如果每次都要自己重新走一遍，那写东西这件事很快就会重新变得麻烦。

所以后来我又专门做了一个用来发博客的 agent，并且给它做了一个博客发布的 skill。

如果要打个不那么技术的比方，**skill 就像把一整套流程封装好了**。原来那些要自己记住、自己串起来的动作，现在被收成了一条可以稳定复用的工作流。你写好内容，其他事情交给龙虾去做。

我很喜欢这种变化。不是因为它听起来多高级，而是因为它真的让“发出去”这件事变简单了。以前你面对的是一串技术步骤，现在你面对的更像是一个清楚的意图：我想把这篇发出去。剩下那些重复、机械、容易出错的动作，不必每次再由人亲自重新走一遍。

## 我想起了我尘封的微信公众号

博客发顺以后，我又想起了很久没怎么动过的微信公众号。

既然内容都已经写好了，博客也已经发了，那能不能顺手同步到公众号？这件事以前听起来很像一句“以后再说”，因为只要一想到公众号 API、封面图、正文图片、排版转换这些东西，就很容易先觉得麻烦。

但这次我没有让它停在“想想而已”。

我又做了一个同步微信公众号的 skill。逻辑其实和博客发布那条线一样：把原来零碎、技术味很重的一整串操作，收成一条能复用的工作流。同样的内容，博客这边发完以后，再通过公众号 API 走一遍同步流程。封面图怎么处理，正文图片怎么上传，阅读原文怎么放，最后怎么生成公众号草稿，这些事情也被一点点磨顺了。

最后形成的状态，是我很喜欢的那种：它先帮我把内容同步成公众号草稿，我再做一点点微调，就可以发。这样既保留了自动化带来的省事，也保留了人最后收口的空间，不会因为图方便，就把表达本身也一起外包出去。

## 技术让应用门槛在降低，但表达能力依然重要

这段时间我越来越强烈的一个感受是，技术并没有消失，但技术的应用门槛，确实在降低。

以前很多事情，默认只有技术人员才比较容易做到。不是因为别人没有需求，也不是因为别人学不会，而是因为从想法到落地之间，横着太多工具、接口、配置、脚本和错误处理。很多人不是不会，而是还没开始，就已经觉得太麻烦。

像 OpenClaw 这样的工具，正在把很多原本必须自己硬啃的技术动作，变成一种可以被调用的能力。于是一些过去只有技术人员才更容易做到的事，开始变得普通人也有机会做到。你不一定要自己写完整脚本，也不一定非得先把所有底层细节都学明白，才能把一件事做起来。

但这不等于表达能力就不重要了。恰恰相反，我现在越来越觉得，有了这样的工具以后，表达能力反而更重要。

因为你要让 OpenClaw 理解你，本身就是一种能力。你得知道自己想要什么，得能把需求说清楚，得知道哪里该收、哪里该放，得在它跑偏的时候把它拉回来，得把一个模糊念头慢慢磨成一个稳定流程。技术门槛在降低，不代表一切都自动完成了；只是以前更重要的是“会不会做”，现在越来越重要的是“知不知道自己要做什么，以及能不能把它说清楚”。

如果说这段时间我真正做成的，不只是一个博客，也不只是两个 skill。那更接近的说法也许是：很多原本只有技术人员才有的能力，正在开始变成普通人也能借助工具抵达的能力。

当然，前提还是那个前提——你得会表达。
